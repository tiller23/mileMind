## 1. Executive Summary

MileMind is an AI-powered training optimization platform that uses a multi-agent architecture to generate, validate, and adapt structured training plans for runners of all levels and distances. Whether training for a first 5K or a 100K ultra, the system grounds all physiological calculations in a deterministic domain layer, ensuring accuracy and preventing hallucinated metrics, while leveraging Claude as the intelligence layer for planning, review, and natural language interaction.

The platform serves two purposes: a genuinely useful personal training tool for runners, and a portfolio-grade demonstration of production agentic AI systems including multi-agent orchestration, tool-calling, self-correction loops, evaluation harnesses, and closed-loop feedback adaptation.

### 1.1 Elevator Pitch

MileMind is an intelligent running coach that instantly generates physiologically perfect, adaptive training plans grounded in real exercise science and lets you negotiate changes conversationally while two AI agents self-correct in the background to keep every workout safe and effective.

### 1.2 Refined Positioning

For everyday runners and seasoned marathoners alike, MileMind is the most trustworthy personal training tool on the market. It responds to the chaos of real life without ever hallucinating physiological metrics, preventing injury while maximizing progress. Simultaneously, MileMind serves as a masterclass portfolio demonstration of production-grade agentic AI, showcasing multi-agent orchestration, deterministic tool-calling, self-correction loops, and quantitative evaluation harnesses in a real-world application.

## 2. Product Vision and Key Differentiators

### 2.1 Vision

Build an intelligent training coach that combines the rigor of exercise science with the adaptability of AI agents. Unlike static training plans or simple chatbots, MileMind uses a planner-reviewer agent architecture with deterministic tool-calling to produce training plans that are both physiologically sound and responsive to real-world athlete feedback. The initial focus is running across all distances, from 5K to marathon and ultra, with multi-sport support planned for future iterations.

### 2.2 Key Differentiators

- **Deterministic Grounding:** All physiological metrics are computed by validated pure-Python models, never generated by the LLM. While competitors let an LLM guess what a safe mileage increase is, MileMind's LLM only negotiates the schedule—the math (Fitness-Fatigue, ACWR, TSS) dictates the safety. This is a massive trust signal for injury-prone or data-driven runners.
- **Multi-Agent Self-Correction:** Planner and reviewer agents iterate until plans meet safety and progression constraints. No other consumer fitness app currently markets a multi-agent self-correction loop. The narrative that a secondary Reviewer agent actively critiques and forces revisions on the Planner agent before the user ever sees the schedule mimics the dynamic of an elite coaching staff rather than a single chatbot.
- **Natural Language Negotiation:** Athletes can request plan adjustments conversationally, and the system re-validates every change through the full agent loop automatically.
- **Glass-Box Transparency:** Power users and tech-forward athletes can inspect the full agent reasoning chain via a debug view showing exact tool calls, retry loops, and reviewer critiques. This builds unparalleled trust with data-driven runners.
- **Quantitative Evaluation:** A synthetic athlete test harness provides measurable reliability benchmarks, transforming MileMind from a demo project into a production-grade system.
- **Enterprise-Grade Engineering:** The platform is positioned for users who expect high-end, precise engineering—a high-performance optimization engine built for those who treat their training data with rigor.

## 5. System Architecture

The system is organized into six distinct layers, each with clear responsibilities and interfaces. This separation ensures testability, debuggability, and clean boundaries between deterministic computation and AI reasoning.

### 5.1 Architecture Overview

| Layer | Responsibility | Technology |
|-------|---------------|------------|
| Deterministic Domain | Physiological models, math, validation | Pure Python (no AI) |
| Tool Layer | Wraps domain models as callable tools | Python functions with JSON schemas |
| Planner Agent | Proposes training plans using tools | Claude API with tool-use |
| Reviewer Agent | Evaluates plan safety and progression | Claude API (separate instance) |
| Orchestration | Retry loop, convergence tracking | Python (raw API calls) |
| Feedback Loop | Compares predicted vs actual, triggers replan | Python + PostgreSQL |

### 5.2 Deterministic Domain Layer

This is the foundation of the entire system. All physiological calculations happen here in pure Python with no AI involvement. This prevents the LLM from hallucinating metrics and ensures reproducible, testable results. The UI must constantly reinforce that math, not AI, is generating the numbers using subtle UI badges such as "Metrics verified by Deterministic Domain Layer."

Core models include:

- **Fitness-Fatigue Model (Banister-style):** Tracks cumulative training load and recovery using impulse-response functions with configurable time constants (τ1 for fitness decay over days to weeks, τ2 for fatigue decay over hours to days). Computes CTL (chronic training load / fitness), ATL (acute training load / fatigue), and TSB (training stress balance / form).
- **VO2max Estimation and Pace Zone Calculation:** Derives training zones from recent performance data using established Daniels-Gilbert VO2 expenditure equations and Karvonen heart rate formulas. Translates VDOT scores into specific training paces for Recovery, Aerobic, Threshold, and VO2max zones.
- **Weekly Load and Injury Risk Heuristics:** Enforces configurable weekly mileage increase limits (default 10%), monitors acute-to-chronic workload ratio (ACWR) using both rolling average and EWMA variants, and flags risky progressions. Athletes with prior training history can override default thresholds via the athlete preference system, acknowledging increased risk.
- **Taper Decay Modeling:** Models fitness retention and fatigue dissipation during pre-race taper periods. Built naturally into the Banister impulse-response model by setting future training impulse to zero and letting the curves decay according to their respective time constants.
- **Monte Carlo Race Simulation Engine:** Runs stochastic simulations of race-day performance based on current fitness state, fatigue levels, course elevation profile, and environmental factors. Samples runner paces from distributions and returns predicted finish-time ranges with confidence intervals.

### 5.3 Tool Layer

Each deterministic model is wrapped as a callable tool with a well-defined JSON schema that Claude can invoke. The LLM must use these tools rather than free-generating any physiological metrics.

| Tool | Input | Output |
|------|-------|--------|
| compute_training_stress() | Workout type, duration, intensity | TSS/TRIMP score, load classification |
| evaluate_fatigue_state() | Athlete ID, date range | CTL, ATL, TSB, recovery status |
| validate_progression_constraints() | Proposed weekly plan | Pass/fail with violation details |
| simulate_race_outcomes() | Athlete state, race params | Distribution of predicted finish times |
| reallocate_week_load() | Current plan, constraints, swap request | Adjusted plan with revalidated loads |

### 5.4 Agent Architecture

#### 5.4.1 Planner Agent

The planner agent is responsible for generating training plans by reasoning about the athlete's goals, current fitness state, and constraints, then calling deterministic tools to compute and validate the plan.

Planner workflow:
- Receives athlete profile, goal event, and current training state
- Proposes a macrocycle structure (base, build, peak, taper phases)
- Generates weekly workout breakdowns with specific sessions
- Calls compute_training_stress() for each proposed workout
- Calls validate_progression_constraints() on the full week
- Iterates on constraint failures before submitting to reviewer

Critical constraint: The planner cannot finalize any plan without passing tool-based validation. Free-generated metrics are rejected by the orchestration layer.

#### 5.4.2 Reviewer Agent

A separate Claude instance that evaluates proposed plans for safety, progression logic, and overall quality. The reviewer has no knowledge of the planner's reasoning process and evaluates the plan independently.

Reviewer evaluates:
- **Safety:** Are rest days adequate? Is intensity distribution appropriate (80/20 rule)?
- **Progression:** Does weekly load increase follow safe patterns? Are step-back weeks included?
- **Specificity:** Do workouts align with the goal event and current training phase?
- **Feasibility:** Are workout durations and intensities realistic for the athlete's level?

#### 5.4.3 Orchestration and Retry Loop

The orchestration layer manages the planner-reviewer interaction loop. If the reviewer rejects a plan, the critique is fed back to the planner along with the specific violations. The planner then re-runs tool calculations and revises the plan.

Convergence parameters:
- **Maximum retry attempts:** Configurable (default: 5)
- **Convergence criteria:** All safety constraints pass, reviewer approval score above threshold
- **Logged metrics:** Retry count, failure reasons, time to convergence, token usage per iteration
- **Fallback behavior:** If max retries reached, surface last best plan with warnings to user: "We had to pull back your mileage slightly to guarantee progression safety."

### 5.5 Feedback Adaptation Loop

After athletes log completed workouts, the system compares predicted training stress against actual performance data. Significant deviations trigger automatic replanning to keep the training plan responsive to real-world conditions.

Deviation triggers:
- Actual training stress deviates more than 15% from predicted
- Athlete reports illness, injury, or missed workouts
- Heart rate or pace data suggests overtraining or under-recovery
- Athlete requests plan modification through chat negotiation
- Wearable recovery data (e.g., WHOOP red recovery day) indicates compromised readiness

### 5.6 Athlete Preference and Override System

Not all athletes are the same, and rigid rules can be counterproductive for experienced runners. MileMind allows athletes to customize progression constraints and training preferences, giving them ownership over their plan while maintaining safety guardrails.

Configurable athlete preferences:
- **Weekly mileage increase limit:** Default 10%, adjustable up to 20% for experienced runners with established base mileage
- **Intensity distribution:** Default 80/20 easy-to-hard ratio, adjustable based on athlete preference and training philosophy
- **Rest day frequency:** Minimum rest days per week, configurable per athlete
- **Long run cap:** Maximum percentage of weekly mileage in a single run
- **Risk tolerance:** Conservative, moderate, or aggressive progression profiles

When an athlete overrides default constraints, the system acknowledges the override, warns about increased risk where appropriate, and adjusts the reviewer's evaluation criteria accordingly. The override is logged so the feedback adaptation loop can track whether aggressive overrides correlate with negative outcomes over time. Hard safety limits (such as ACWR above 1.5) remain enforced regardless of athlete preference.

## 6. User Experience

### 6.1 Design Principles

Runner-centric, trustworthy, and heavily data-grounded. Use a dark-mode default palette (deep slates, vibrant electric lime for pacing zones, and crisp white typography). Mobile-first and on-the-run: large tap targets, swipe-to-log mechanics, and voice-to-text prioritized for the chat interface.

### 6.2 Key User Journeys

#### 6.2.1 Journey 1: Onboarding and Baseline Setup

- User opens the app; greeted by a clean, motivational value proposition
- Authenticates via OAuth (Apple/Google/Strava)
- Completes a conversational wizard: goal distance/date, current fitness, injury history, and risk tolerance
- Connects data sources (Apple Health/Strava) to compute an immediate deterministic baseline (CTL/VO2max)

**Success Criteria:** >90% onboarding completion; valid baseline Fitness-Fatigue state established.

#### 6.2.2 Journey 2: First Plan Generation (The Agent Loop)

- User confirms goals. UI displays an engaging "Agents Collaborating" loading state with code blocks and metrics
- Backend: Planner generates macrocycle, calls Python tools, Reviewer checks safety constraints, retries if necessary
- Plan is finalized and pushed to the frontend with predicted finish-time range

**Success Criteria:** End-to-end generation in <30 seconds; 0% safety constraint violations.

#### 6.2.3 Journey 3: Natural Language Plan Negotiation

- User taps the floating Coach Chat bubble
- Inputs via voice or text (e.g., "I tweaked my knee, can we swap tomorrow's track workout for something low impact?")
- Backend: Planner triggers reallocate_week_load(), adjusting the schedule. Reviewer validates the new ACWR
- UI highlights the changed calendar days with accept/reject options

**Success Criteria:** User accepts the modification on the first or second turn >85% of the time.

#### 6.2.4 Journey 4: Daily Logging and Feedback Loop

- Post-run, data syncs automatically from connected devices
- User receives a rich notification: "Workout complete! How did it feel?"
- User inputs RPE (1–10) and notes
- Backend: If actual TSS deviates >15% from predicted, the adaptation loop silently recalculates the next 3 days

**Success Criteria:** High daily engagement; auto-adaptation triggers correctly on significant deviations.

#### 6.2.5 Journey 5: The Debug/Transparency View

- User long-presses a workout card and selects "Show AI Brain"
- A console-like overlay appears showing exact Claude prompts, deterministic tool outputs, and Reviewer critique
- Displays full iteration history, JSON payloads, tool-call latency metrics, and token usage

**Success Criteria:** Flawless rendering of tool outputs and iteration history; serves as the primary wow factor for technical demos.

### 6.3 Dashboard Components

- **Weekly Calendar View:** Structured training plan displayed as a calendar with individual workout cards showing type, distance, pace zone, estimated duration, and training stress
- **Training Load Chart:** Beautiful, interactive line chart displaying CTL (fitness, rising steady), ATL (fatigue, spiking after long runs), and TSB (form, dipping and recovering). Future projections shown as a shaded area based on the planned macrocycle
- **Workout Detail View:** Expanded view of each session with prescribed paces, warm-up/cool-down structure, and purpose explanation
- **Pace Zone Pills:** Recovery, Aerobic, Threshold, and VO2 zones represented as colored pills
- **Progress Metrics:** Weekly mileage trends, workout compliance rate, predicted race performance
- **Event Countdown:** Days to goal race with current readiness assessment / Race Day Readiness Widget
- **Debug View:** Designed like a sleek developer terminal with monospace fonts for JSON payloads and tool-call latency metrics

### 6.4 Chat Negotiation Panel

A conversational interface embedded in the dashboard that allows athletes to negotiate plan adjustments with the planner agent. This creates a natural interaction model where users can request changes in plain language and the system re-validates automatically.

Example interactions:
- "Move my long run to Saturday, I have work late on Friday"
- "This week feels too hard, can you pull back the intensity?"
- "I felt great on today's tempo run, can we be more aggressive next week?"
- "I tweaked my knee, can you adjust the plan to reduce impact?"

Each negotiation triggers the full planner-reviewer loop: the planner proposes a modification, calls reallocate_week_load() to recompute stress, validates constraints, and the reviewer approves or requests further revision. The user sees only the final adjusted plan unless they enable the debug view.

### 6.5 Authentication and Multi-User

The platform supports multiple user accounts from day one. Each athlete has their own profile, training history, and plan state. Authentication is handled via standard email/password or OAuth (Google/Apple/Strava) with JWT session management.

## 7. Technical Stack

### 7.1 Stack Overview

| Component | Technology | Rationale |
|-----------|------------|-----------|
| Backend | Python (FastAPI) | Native support for scientific computing (NumPy/SciPy), clean async API layer, high-performance for deterministic math and agent orchestration |
| Agent Orchestration | Raw Anthropic API with tool-use | Full control over retry loop, tool execution, and deterministic boundaries. No heavy framework abstractions; portfolio-friendly |
| LLM (Planning) | Claude Sonnet (cost-optimized) | Cheaper model for iterative planning loops; handles 90% of planning work |
| LLM (Review) | Claude Opus (fallback/complex) | Higher capability for safety-critical review decisions on advanced marathon/ultra plans |
| Frontend (Mobile) | React Native (Expo) | Rapid cross-platform iOS/Android deployment for mobile-first experience |
| Frontend (Web) | React / Next.js | Industry standard; detailed dashboard views, deeper data analysis, seamless Vercel deployment |
| Database | PostgreSQL | Relational structure for plans, workouts, profiles; JSONB columns for flexible plan/workout storage |
| Data Integration | Strava API (v3) | Workout import, activity sync, broad device compatibility |
| Deployment | Cloud-hosted (live URL) | Publicly accessible for demos and real-world use |
| Dev Workflow | Claude Code | AI-assisted development for accelerated build velocity |

### 7.2 API Contracts

Key API endpoints:
- **POST /api/v1/plans/generate:** Initiates the agent loop for full macrocycle creation
- **POST /api/v1/plans/{id}/negotiate:** Accepts natural language string; returns modified plan JSON
- **GET /api/v1/plans/{id}/debug_trace:** Returns the full iteration and tool-call history

### 7.3 Model Cost Optimization

The system uses a tiered model strategy to manage API costs. The planner agent, which may iterate multiple times per plan generation, uses a cost-optimized model (Claude Sonnet). The reviewer agent, which makes safety-critical decisions, uses a higher-capability model (Claude Opus) when needed. Simple validation-only passes can use the cheaper model for both roles. Claude Sonnet handles 90% of the iterative planning; Opus is invoked only for final safety sign-off on advanced marathon/ultra plans.

### 7.4 Database Schema

| Entity | Key Fields | Relationships |
|--------|------------|---------------|
| User | id, email, auth_provider, created_at | Has many Athletes |

| Training Plan | id, athlete_id, goal_event, start_date, end_date, status | Belongs to Athlete, has many Weeks |

## 8. Evaluation Harness

The evaluation harness is a critical differentiator that transforms MileMind from a demo project into a production-grade system with quantifiable reliability metrics. It creates synthetic test athletes with known characteristics and measures system performance across multiple dimensions.

### 8.1 Synthetic Test Athletes

| Persona | Profile | Expected Behavior |
|---------|---------|-------------------|
| Beginner Runner | 5–8 mpw, no race history, low VO2max | Very conservative progression, walk-run intervals, extra rest days, no speed work early |
| Overtrained Athlete | High chronic load, negative TSB, fatigue markers | Immediate load reduction, recovery focus, gradual rebuild |
| Aggressive Spiker | Requests 40% weekly mileage increase | System should reject and propose safe alternative |
| Injury-Prone Runner | History of shin splints, recent IT-band issue | Low-impact alternatives, reduced intensity, cross-training emphasis |
| Advanced Marathoner | 60+ mpw, multiple race PRs, high VO2max | Sophisticated periodization, race-specific workouts, fine-tuned taper |

### 8.2 Tracked Metrics

| Metric | Description | Target |
|--------|-------------|--------|
| Constraint Violation Rate | Percentage of generated plans that violate safety constraints | < 5% after reviewer loop |
| Average Retries to Convergence | Mean iterations before reviewer approval | < 3 retries |
| Plan Safety Score | Composite score of load progression, rest adequacy, intensity distribution | > 85/100 |
| Overload Frequency | How often plans exceed safe ACWR thresholds | < 2% of planned weeks |
| Execution Time | End-to-end plan generation latency | < 30 seconds for weekly plan |
| Token Efficiency | Total tokens consumed per successful plan generation | Tracked, optimized over time |

### 8.3 Benchmarking Process

Each synthetic athlete is run through the full plan generation pipeline repeatedly to measure variance and reliability. Results are stored in PostgreSQL and can be visualized on the dashboard. Regression testing ensures that system updates do not degrade performance on established test cases. The Synthetic Test Athlete evaluation harness runs as a CI/CD pipeline step. Any changes to the deterministic math must pass 100% of the persona benchmarks.

Initial development and testing uses programmatically generated athlete profiles and workout histories. This allows rapid iteration on the deterministic engine and agent orchestration without external API dependencies.

### 9.2 Phase 2: Strava API

Integration scope:

To push beyond standard integrations and leverage cutting-edge recovery data:

- **Eight Sleep:** Sleep environment and recovery data for comprehensive recovery modeling.

If a user wakes up with a severely compromised recovery score from any connected wearable, the feedback loop should proactively suggest converting a planned threshold interval session to an easy Zone 2 recovery spin.

## 10. Data, Privacy, Safety, and Trust

### 10.1 Data Grounding for Trust

The UI must constantly reinforce that math, not AI, is generating the numbers. Use subtle UI badges: "Metrics verified by Deterministic Domain Layer." The system prompt enforcement, output validation layer, and strict rejection of responses with un-sourced numbers ensure complete safety.

### 10.2 Health Data Compliance

Direct integration with Apple HealthKit requires strict privacy scopes. Data must be encrypted at rest, with a clear one-tap "Delete My Data" option. All health data processing follows privacy-by-design principles.

### 10.3 Safety Overrides

- **Medical Disclaimer:** Prominent onboarding banner: "MileMind is an optimization tool, not medical advice."
- **Hard Limits:** Even if a user sets their preference to Aggressive, the backend pure-Python logic must hard-cap the Acute-to-Chronic Workload Ratio (ACWR) at 1.5. If the LLM tries to bypass this, the Python execution layer rejects the tool call immediately.
- **Validation Layer:** The Tool Layer strictly validates all outputs against JSON schemas. Any un-sourced number in the Planner's final payload is caught by the pure-Python orchestrator, triggering a hard fail/retry before the user ever sees it.

## 11. Competitive Landscape

The AI running coach space is experiencing a massive surge in 2025 and 2026. MileMind enters a highly competitive but rapidly evolving market divided into distinct categories.

### 11.1 Market Categories

- **The Polished Giants (Runna, Coopah):** Dominate the consumer space with highly refined UI/UX, structured plans, and lifestyle flexibility. However, their AI is primarily a mix of traditional decision trees for onboarding and basic LLM wrappers for customer support chat.
- **The AI Chatbots (NXT RUN, Racemate, The Running Genie):** Heavily market conversational AI. NXT RUN allows users to toggle between Claude, Gemini, and GPT. They parse Strava data and allow chat about workouts, but rely entirely on the LLM to generate training advice—risking hallucinated unsafe mileage jumps.
- **The Hyper-Adaptive Engines (HumanGO, TrainAsONE):** Focus intensely on adjusting schedules dynamically based on performance deviations, fatigue, and availability. Highly algorithmic but interfaces can feel overly analytical or rigid.

### 11.2 MileMind's Structural Advantage

Most current AI running apps suffer from the "wrapper" problem: they take a static template and use an LLM to dynamically tweak it. MileMind's architecture solves this by separating the concerns: the LLM only negotiates the schedule while the deterministic math dictates safety. The multi-agent peer review and glass-box transparency have no equivalent in the current consumer fitness market.

### 11.3 Key Competitor Features to Adopt

- **B-Race Integration (Runna/Coopah):** Runners rarely train for just one race. The system must seamlessly handle secondary races mid-macrocycle, automatically calculating the minor taper and recovery required without derailing the main plan.
- **Deep Recovery Wearable Sync:** Integrate directly with WHOOP, Oura, or Eight Sleep for proactive recovery-based adjustments.
- **Race Day Readiness Widget:** Translate the Monte Carlo simulation into a highly visible, daily updated readiness metric (akin to Coopah's Race Day Confidence Score, but vastly more sophisticated).
- **Strength and Cross-Training:** Running-only plans create a gap. The deterministic engine should eventually account for the TSS of leg days or plyometrics to accurately model total CNS fatigue.
- **Pre-Run Audio Briefings:** Allow the Planner agent to generate a dynamic, 15-second personalized audio briefing before a run to bridge the gap between digital planning and physical execution.

## 12. Engagement, Retention, and Monetization

### 12.1 Retention Hooks

- **Sunday Evening Lookahead:** A push notification summarizing the upcoming week's goals and predicted physiological adaptations.

## 13. Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| LLM ignores tools, generates metrics | High – unsafe plans | System prompt enforcement, output validation layer, reject responses with un-sourced numbers. The pure-Python orchestrator triggers a hard fail/retry before the user ever sees it |
| Deterministic models produce incorrect values | High – unsafe plans | Extensive unit testing against published exercise science, manual review of edge cases, 100% test coverage on core calculations |
| Agent loop fails to converge (infinite loops) | Medium – degraded UX | Hard cap at 5 retries. If unresolved, surface last safest plan with UI warning. Graceful fallback with warnings |
| API costs exceed budget | Medium – sustainability | Tiered model strategy (Sonnet for 90%, Opus for safety sign-off), caching, token usage monitoring, cost alerts |
| Strava API rate limits or changes | Low – data integration delay | Graceful degradation to manual entry, webhook-based sync to minimize polling |
| Scope creep beyond MVP | High – never ships | Strict phase gating; each phase has defined success criteria before moving to next |
| Test harness drift | Medium – silent regressions | The Synthetic Test Athlete evaluation harness runs as a CI/CD pipeline step. Any changes to deterministic math must pass 100% of persona benchmarks |

## 16. Open-Source Reference Implementations

No single repo contains the exact full set of models specified in the deterministic domain layer, but a production-grade Python module can be assembled by combining the best pieces from established open-source projects. All referenced repositories are MIT/Apache/GPL or similar licensed.

### 16.1 Fitness-Fatigue Model (Banister)

- **GoldenCheetah:** The gold standard open-source desktop app for runners. Implements the Banister impulse-response model, PMC charts, CTL, ATL, TSB, TRIMP, and taper/fatigue decay. Core math is in C++ (src/Engine/ and src/Metrics/), easily portable to Python. github.com/GoldenCheetah/GoldenCheetah
- **choochoo SHRIMP:** Python-friendly Banister variant implementing scaled heart-rate impulse response with configurable time constants. andrewcooke.github.io/choochoo/impulse.html
- **intervalsicugptcoach-public:** Explicitly references and uses Banister Fitness-Fatigue (CTL/ATL/TSB) in its coaching framework. github.com/revo2wheels/intervalsicugptcoach-public

### 16.2 VO2max / Pace Zone Calculation

- **mattncott/running-calculations:** TypeScript but 100% pure math—trivial to convert to Python. Includes calculateVDOT, calculatePercentageVO2Max, calculateZonesFromFTP, Daniels tables, TSS. github.com/mattncott/running-calculations
- **mekeetsa/vdot:** Direct implementation of Daniels-Gilbert VO2 expenditure equations from Oxygen Power and Daniels' Running Formula. github.com/mekeetsa/vdot
- **st3v/running-formulas-mcp:** VDOT plus all Daniels training paces. github.com/st3v/running-formulas-mcp

### 16.3 ACWR / Injury Risk

- **ale-uy/Acute_Chronic_Workload_Ratio:** Full Python ACWR with rolling averages and EWMA variants—exactly the injury-risk monitoring required. github.com/ale-uy/Acute_Chronic_Workload_Ratio

### 16.4 Monte Carlo Race Simulation

- **mountain-software-jp/trail-simulator:** Pure Python Monte Carlo for trail/ultra races using GPX courses. Samples runner paces from normal distribution, simulates congestion, cutoffs, elevation effects. Foundation for the race simulation engine—just add Banister-derived fitness state as the mean pace input. github.com/mountain-software-jp/trail-simulator

### 16.5 Recommended Assembly Order

- Start with mattncott/running-calculations → convert the TS math to Python (1–2 hours)
- Add Banister/CTL-ATL-TSB from GoldenCheetah or choochoo (1–2 days)
- Drop in ale-uy ACWR for risk heuristics
- Wrap the Daniels VDOT/paces from mekeetsa/vdot or st3v
- Extend the trail-simulator Monte Carlo with current athlete state

This produces a clean, testable, pure-Python deterministic_domain.py module that matches the architecture specification exactly and is fully unit-testable as required in Phase 1.
